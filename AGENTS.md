### Aurora PostgreSQL integration rules

- **Secrets**: NEVER hardcode DB credentials or commit connection strings. In dev, read them from env (GitHub Actions/Vercel secrets); in prod, fetch from AWS Secrets Manager with managed rotation enabled, and re-fetch on auth failure rather than caching forever — rotation changes the password under you.
- **Prefer IAM auth in prod**: connect with short-lived RDS IAM auth tokens (15-min lifetime — regenerate per connection, never cache) over the `rds_iam` PostgreSQL role; fall back to a Secrets Manager password only where IAM auth isn't viable.
- **Least privilege**: give the app role only the rights it needs (no superuser/`rds_superuser`); use a separate, narrower role for read-only/reader-endpoint queries and for migrations.
- **TLS is mandatory**: always connect with `sslmode=verify-full` and the Amazon RDS CA bundle (`global-bundle.pem`) as `sslrootcert`; set the cluster parameter `rds.force_ssl=1`. NEVER use `require`/`prefer`/`disable` or `rejectUnauthorized:false`. Ship the CA bundle with the app and track CA rotation.
- **Connections**: from short-lived contexts (GitHub Actions jobs, Next.js serverless/route handlers) do NOT open raw per-invocation pools to the cluster — route through RDS Proxy (with IAM auth) or the RDS Data API for Serverless v2 to avoid connection storms. Route writes to the cluster/writer endpoint and read-only queries to the reader endpoint.
- **Failover-safe DNS**: set DNS TTL < 30s; on transient connection errors (failover, resume-from-scaling, too-many-connections) retry with exponential backoff **and jitter**. Treat post-failover errors as "unknown outcome" — never assume an in-flight write didn't land.
- **Idempotent writes**: every insert that can be retried (LangGraph node re-runs, queue redelivery, failover retry) MUST carry a unique idempotency/dedup key and use `INSERT ... ON CONFLICT ... DO NOTHING/UPDATE`; wrap multi-statement steps in a single transaction so a retry can't double-apply.
- **Event "webhooks" (RDS events via SNS → Next.js routes)**: verify every inbound SNS message signature before acting — confirm `SigningCertURL` host ends in `.amazonaws.com` (an SNS endpoint) over HTTPS, prefer `SignatureVersion` 2 (SHA256), validate against the cert's public key, and reject unexpected `TopicArn`s. Handle `SubscriptionConfirmation` explicitly. NEVER trust the payload before signature verification.
- **Replay protection**: dedupe on SNS `MessageId` (persist processed IDs) so redelivered RDS event notifications are processed at-most-once; keep sandbox and prod on separate SNS topics + secrets.
- **pgvector / data hygiene**: redact PII and secrets from evidence BEFORE it is written to Aurora or embedded into pgvector — embeddings and stored rows are hard to scrub later. Run schema changes via versioned migrations, never ad-hoc DDL.

### GDPR compliance & data-safety rules

- **NEVER log, print, or persist raw personal data.** Redact/pseudonymize PII (names, emails, IPs, user IDs, free-text that may contain it) *before* it enters LangGraph state, Aurora rows, pgvector embeddings, S3 evidence bundles, telemetry, or any log line. Logs are processing under GDPR — treat them like a database.
- **Minimize at the source.** Collect and store only fields the feature actually needs; never `SELECT *` PII into model context or evidence. Default to dropping a field rather than carrying it "just in case."
- **Set retention/TTL on everything that can hold PII** — logs, evidence bundles, embeddings, demo media, and backups. No unbounded retention; define and enforce a deletion policy per data class.
- **Pseudonymization ≠ anonymization** (EDPB Guidelines 01/2025): pseudonymized data is still personal data and stays in GDPR scope. Store the re-identification mapping/keys separately from the data, behind strict access control — never in the same table, repo, or log stream.
- **Run Bedrock Guardrails PII filters on both Converse inputs and outputs**, plus deterministic redaction checks, so personal data is stripped before generation and never surfaces in generated artifacts or claim provenance.
- **Keep PII off the client.** No personal data in React Server→Client props, URL paths/query params, `localStorage`/`sessionStorage`, hidden DOM, or third-party analytics/error trackers. Gate any PII fetch behind server-side authz.
- **Encrypt PII in transit (TLS) and at rest (e.g. AES-256).** Never disable TLS verification to Aurora/Bedrock/S3, even in dev.
- **Honor data-subject rights in the data model.** Design deletes so erasure (Art. 17) cascades to Aurora rows, pgvector embeddings, S3 objects, and derived artifacts — not just the primary record. Support access/export (Arts. 15, 20) as machine-readable output. Erasure in backups is a known enforcement gap (EDPB CEF 2025) — document how it's handled.
- **No non-essential cookies, tracking, or fingerprinting without prior opt-in consent.** Consent must be freely given, specific, granular, and as easy to withdraw as to give; strictly-necessary only by default. Never wall the app behind consent.
- **Establish and record a lawful basis + purpose** for each new personal-data flow; do not reuse data collected for one purpose in an unrelated one (purpose limitation).
- **Use synthetic/fixture data in Playwright** runs, screenshots, traces, and videos — real PII must never land in test artifacts, demo captures, or CI logs.
- **Return user-safe error messages**; never echo personal data, stack internals, or raw evidence into error payloads, API responses, or client toasts.

### Security baseline

- NEVER hardcode secrets (API keys, tokens, passwords). Use env vars or a secret manager; read them at startup.
- Validate and sanitize ALL external input. Use parameterized queries (no string-built SQL); escape output to prevent XSS.

- Apply least privilege: the narrowest scope/permission/token that works.
- Never log secrets, credentials, or regulated/PII data. Keep error messages free of sensitive detail.

- Pin and audit dependencies; treat a new dependency as new attack surface.

### Vercel integration rules

- **MUST verify every inbound Vercel webhook/log-drain request** by comparing the `x-vercel-signature` header against an `HMAC-SHA1` hash of the **raw request body**; reject with `403` on missing/mismatched signatures. Use a constant-time compare (`crypto.timingSafeEqual`), never `===`.
