# Product Context

Ground every generated artifact in what the product actually is and who it serves, so
claims stay accurate and on-message.

- State the product's purpose in one plain sentence the reader would recognize.
- Describe capabilities in terms of the user outcome they unlock, not the internal
  component that implements them.
- Use the product's released, public name for every feature; never surface an internal
  codename, service name, or unreleased feature flag (see [[redaction-rules]]).
- Tie each feature back to the concrete problem it solves; if the supporting evidence
  does not establish the problem, omit the framing rather than invent it.
- Defer voice and phrasing decisions to [[brand-voice]] and audience framing to
  [[audience-map]]; this skill owns *what the product is*, not *how it sounds*.
