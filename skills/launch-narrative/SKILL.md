# Launch narrative

A release ships across many touchpoints — blog, changelog, social, email, demo. They must tell
ONE coherent story, not N disconnected posts. Before drafting any single artifact, fix the
release's spine and reuse it everywhere.

## The spine (derive once per release, reuse across every artifact)

- **Headline change** — the single most important thing a user can now do.
- **Who it's for** — the primary ICP segment(s) this serves.
- **Before → after** — the friction that existed, and what's different now.
- **Proof** — the one concrete, evidence-backed capability that makes it credible.
- **One call to action** — the next step (try it, read the docs, book a demo), consistent everywhere.

## How to apply it per channel

- Lead with the same headline change in every artifact; adapt depth and tone to the channel, never the story.
- Name the feature the same way everywhere — consistent terminology across channels.
- Sequence, don't repeat: the blog is the canonical narrative, social teases it, email routes to it, the changelog records it precisely.
- Never introduce a claim or angle in one channel that another channel contradicts.

## Anti-patterns

- Each artifact inventing its own framing or feature name.
- A social post over-promising what the blog or changelog actually supports.
- Multiple competing calls to action across channels.
